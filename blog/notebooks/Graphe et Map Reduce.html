Graphes et Map Reduce¶

On suppose qu'on a un graphe \(G(V,E)\) pour lequel on cherche dterminer la distance de tous les noeuds un noeud prcis du graphe. Si calculer cette distance pour un graphe de petite taille ne pose pas de problme, pour un grand graphe, cela peut se rvler plus complexe. L'ide de cet exercice est de montrer que les concepts Map Reduce ne sont pas bien adapts cela. Pour un grand graphe (plusieurs dizaines de millions de noeuds), le calcul est soit trop long faire sur un seul ordinateur, soit cet ordinateur ne peut tout simplement pas contenir tout le graphe. Il faut alors rpartir ce calcul sur plusieurs machines. Mais commenons d'abord par un graphe simple et symtrique : si on peut passer d'un noeud \(v\) \(v'\), le chemin en sens inverse est possible.

Par exemple, les noeuds 0 et 5 sont relis par un arc. Leur distance est donc de 1. Mais tous les noeuds ne sont pas directement relis entre eux. Il faut parfois emprunter deux ou trois arcs pour passer du noeud 0 un autre. On calculer cette distance minimale \(d(v,v')\) o \(v\) et \(v'\) sont deux noeuds du graphe. Par exemple, partons du noeud 0 et explorons de proche en proche le graphe. Au dbut, tout ce qu'on connat est :

\[ \left\{\begin{array}{l} d(0,0) = 0 \\ \forall v, \; d(0,v) = \infty \end{array} \right . \]

Pour viter les critures trop longues, on note par la suite \(d(0,i)=d(i)\). Le calcul est identique quelque soit le noeud. Un algorithme simple consiste mettre jour ce tableau jusqu' ce qu'il n'volue plus. Une mise jour consiste prendre un lment du tableau, par exemple \(d(i)\) et un arc \(d(i,j)\) pour mettre jour \(d(j)\). On sait alors que \(d(j) \leq d(i)+1\).

def enumerate_arcs(G):
    for i,j in G.edges():
        yield i,j
        yield j,i
d = { 0:0 }
maj = 1
while maj > 0:
    maj = 0
    for i,j in enumerate_arcs(G):
        if i in d:
            x = d[i] + 1
            if j in d: 
                if d[j] > x :
                    d[j] = x
                    maj += 1
            else:
                d[j] = x
                maj += 1
labels={ g:"d({0})={1}".format(g,d[g]) for g in G.nodes() }
plt.figure(figsize=(12,4))
nx.draw(G,pos,node_color='#A0CBE2', labels=labels, node_size=2000,font_size=10)
plt.show()

Chaque valeur indique la distance d'un noeud au noeud 0. Dans ce graphe, il faut passer par trois arcs au maximum pour aller du noeud 0 n'importe quel autre noeud.

Map Reduce

L'algorithme prcdent a un cot proportionnel au nombre d'itrations multipli par le nombre d'arcs et l'accs au dictionnaire d, soit au pire \(O(|G||E|\ln_2|G|)\) o \(|x|\) est le nombre d'lments de cet ensemble. Au pire le plus grand chemin ne peut excder le nombre de noeuds. L'autre caractristique de cet algorithme est qu'il fait intervenir deux ensembles : les arcs E=G.edges() et le tableau d. Ce qu'il est important de retenir est que chaque itration requiert l'accs tous les lments du tableau d. Or, pour des grands graphes, d est trop grand pour qu'on puisse accder tous ses lments la fois. Il faut rpartir d sur plusieurs machines.

Lorsqu'on voque Map Reduce, une des technologies open source les plus utilises aujourd'hui est Hadoop et un langage associ PIG. Je ne dtaillerai pas le langage mais il faut retenir que sa logique est proche du SQL et qu'on ne peut pas utiliser d'index. Comment implmenter l'algorithme prcdent en SQL en partant du principe que l'ensemble des arcs \(E\) et du tableau \(d\) sont des tables, qu'on peut les parcourir mais qu'on ne peut plus accder un lment en particulier ?

Il suffit de crer autant de VIEW que d'itrations. Mais ce mcanisme apparat moins souple que la version implmente en langage Python. La premire difficult est le nombre d'itrations qui n'est pas connu l'avance. Avec le langage SQL, ce nombre d'itrations ne peut pas dpendre des donnes. Il en est de mme avec un langage Map Reduce.

En revanche, il est plus facile de parallliser la version SQL sur plusieurs machines car une requte SQL s'y prte bien. Les problmes de paralllisation surviennent gnralement si le graphe est trs connect : l'instruction INNER JOIN a un cot infrieur \(O(|E||G|^2)\). Cette valeur est suprieure au cot de l'algorithme cod en Python car il est ralis sans index. Moins il y a d'arcs, plus c'est rapide.

Cependant, dans les faits, on remarque qu'il suffit qu'un petit nombre de noeuds soient connects beaucoup d'autres pour que des problmes de paralllisation surviennent. En rgle gnrale, chaque machine reoit toutes les donnes qui partagent la mme cl utilise dans l'instruction INNER JOIN. Si la quantit de donnes associe une cl est trs grande, il est possible que la machine traitant les donnes de cette cl termine son excution bien plus tard que toutes les autres.

C'est dans ce cas qu'il faut parfois user d'imagination.
